Project Overview

💡
Core Value Proposition: A single, unified, enterprise-ready LMS portal combining AI-powered assessments, live video sessions, automated payment management, resume ATS scanning, real-time notifications, and a multi-role access-controlled environment — all in one platform.

1.1 Problem Statement

Traditional training institutions face critical bottlenecks in managing large cohorts. The key pain points this system resolves:

  • Fragmented learning delivery — No unified platform for videos, live classes, assessments, and resources
  • Manual administrative overhead — Enrollment approvals, payment tracking, and batch assignments done manually
  • Zero AI integration — Question paper generation, resume scanning, and email automation handled entirely by humans
  • No real-time communication — No live chat, doubt resolution, or real-time notifications between stakeholders
  • Lack of access control — Inability to restrict content per batch, schedule, or role

1.2 Business Objective

Build a full-stack, enterprise-grade, AI-augmented LMS that:

  • Automates course lifecycle management from creation to enrollment to completion
  • Integrates AI for question generation, resume ATS scanning, and broadcast email automation
  • Supports multi-role portals (Student, Instructor, Manager, Admin) with strict RBAC
  • Delivers live classes via Zoom SDK integration
  • Enables real-time communication via Socket.IO

1.3 Target Users

🛡️
Admin
Full platform control; approvals, security, user management. Requires 2FA OTP.
📊
Manager
Oversee batches, generate AI questions, monitor student progress.
🎓

Course creators; upload content, conduct live classes, grade submissions.
📚

Learners enrolled in courses; access videos, exams, and resources.

End-to-End Workflows

5.1 User Registration & Login Flow

Authentication Flow — OTP → Signup → JWT

5.2 AI Resume Scan Flow (ATS)

Resume ATS Flow — Student → n8n AI → Score Card 
Student uploads PDF resume
  → POST /api/user/scan-resume (multipart/form-data)
  → Multer processes file into memory buffer
  → pdf-parse extracts raw text from PDF
  → Text sent to n8n AI webhook (ATS analysis)

n8n Workflow (AI Model)
  → Analyzes resume against job-market criteria
  → Returns: score (0-100), strengths, weaknesses, suggestions

Backend saves ResumeScan document to MongoDB
  → Decrements user's ats_credits (default: 3 free scans)
  → Optionally uploads PDF to Cloudinary for storage

Frontend displays score card with detailed feedback sections

5.3 AI Question Generation Flow

Question Bank Generator — Manager → AI → DB 
Manager/Instructor opens Question Bank Generator
  → Selects: Topic, Question Type, Count, Difficulty
  → Types: MCQ | True/False | Short Answer | Long Answer | Fill-in-Blank | Coding

POST /api/manager/generate-questions
  → Backend routes to correct n8n webhook by question type:
    - MCQ        → N8N_MCQ_GENERATOR_URL
    - True/False → N8N_TRUE_FALSE_GENERATOR_URL
    - Coding     → N8N_CODING_GENERATOR_URL
    (timeout: 120 seconds for AI processing)

n8n AI Workflow → LLM generates questions
  → Questions displayed in UI for Manager review/edit
  → Manager approves → Saved to QuestionBank collection
  → Questions linked to Exam or MockPaper

5.4 Live Class Workflow (Zoom)

Zoom Live Class — Create → Join → Webhook 
Instructor creates Live Class
  → POST /api/zoom/meetings (Server-to-Server OAuth)
  → Backend calls Zoom OAuth: grant_type=account_credentials
  → Access token obtained
  → POST https://api.zoom.us/v2/users/me/meetings
  → Returns: meetingId, joinUrl, startUrl, password
  → LiveClass document saved to MongoDB

Student joins class
  → POST /api/zoom/signature → JWT signed with SDK key/secret
  → Frontend Zoom Meeting SDK initializes in browser
  → Student joins meeting in embedded Zoom component

Zoom Webhook Events
  → meeting.started → LiveClass.status = 'live'
  → meeting.ended   → LiveClass.status = 'ended'
  → Signature verified via HMAC-SHA256 with ZOOM_SECRET_TOKEN

Security Implementation

8.1 Authentication & Authorization Stack

Layer Implementation
Password storage bcrypt with salt factor 10 (industry standard)
Session tokens JWT HS256, 7-day expiry, stored in localStorage
Admin 2FA OTP second-factor required for ALL admin logins
Role enforcement requireRole() middleware on every protected endpoint
Role caching In-memory Map with 30-second TTL to reduce DB hits

8.2 Brute-Force Protection

Failed login attempt → failed_login_attempts++
≥ 3 failures → Profile.approval_status = 'suspended'
SecurityEvent logged with IP address
Admin can manually unsuspend or set suspended_until timer

8.3 Input Validation & Data Protection

Concern Implementation
Request size limit 50MB JSON body limit (for base64 uploads)
NoSQL injection Mongoose schema typing prevents untyped input
XSS protection React's JSX auto-escapes all output
CORS Origin mirroring with credentials support
Zoom webhook security HMAC-SHA256 signature verification
File type enforcement Multer memory storage with type checking
Sensitive field removal password_hash excluded from all JSON responses

8.4 Audit Logging

  • SystemLog collection records: log_type, module, action, user_id
  • SecurityEvent collection records: login failures, IP blocks, suspensions, admin OTP events
  • Admin Security Monitor panel surfaces all security events in real-time

12.2 Generic CRUD Logic (The Model Map)

To handle 25+ collections efficiently, the system uses a Model Mapping strategy in server.js. This reduces code duplication by 80%.

server.js — Generic Data Route Logic 
const MODEL_MAP = {
  'users': User,
  'courses': Course,
  'enrollments': Enrollment,
  // ... maps all 25+ schemas
};

app.get('/api/data/:table', authenticateToken, async (req, res) => {
  const Model = MODEL_MAP[req.params.table];
  if (!Model) return res.status(404).send('Invalid table');
  
  // Apply role-based filtering
  let query = {};
  if (req.user.role === 'student') query.user_id = req.user.id;
  
  const data = await Model.find(query).lean();
  res.json(data);
});

20.2 n8n Integration Logic in Node.js

The backend communicates with n8n using standard Axios POST requests. We use specific environment variables for each workflow URL.

n8n Integration Handler 
async function triggerN8NWorkflow(type, data) {
  const urls = {
    ats: process.env.N8N_ATS_WEBHOOK_URL,
    quiz: process.env.N8N_QUIZ_GEN_URL,
    broadcast: process.env.N8N_EMAIL_BROADCAST_URL
  };
  
  const response = await axios.post(urls[type], data, {
    headers: { 'X-N8N-API-KEY': process.env.N8N_SECRET }
  });
  return response.data;
}
